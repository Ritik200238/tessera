## 1. System Overview

Tessera is three independent runtimes that share one source of truth (the vault contract on Robinhood Chain):

| Runtime | Language | Where it runs | Lifecycle |
|---|---|---|---|
| **Vault contracts** | Rust → WASM (Stylus) + Solidity (oracle, mocks) | Robinhood Chain testnet | Deployed once per environment |
| **Risk Agent** | TypeScript (Vibekit / Node) | Long-running Node process (local for demo, Vercel/Fly later) | One instance per environment |
| **Web UI** | TypeScript (Next.js App Router) | Vercel | Per-deploy |

Coupling is intentionally minimal: the agent and UI both read contract state via JSON-RPC and never share state with each other. The contract is the only consensus point.

```
┌─────────────┐     RPC (read)     ┌────────────────────────┐
│   Web UI    │ ─────────────────► │                        │
│  (Next.js)  │ ◄───── wallet tx ─►│   Robinhood Chain      │
└─────────────┘                    │                        │
                                   │  ┌──────────────────┐  │
┌─────────────┐  RPC (read+write)  │  │  TesseraVault    │  │
│ Risk Agent  │ ◄─────────────────►│  │  (Stylus/Rust)   │  │
│  (Vibekit)  │                    │  └──────────────────┘  │
└─────────────┘                    │  ┌──────────────────┐  │
       │                           │  │  MockOracle      │  │
```

### Why this split

- `contracts/crates/interest-model` is a no-`std`, no-storage crate so the math is unit-testable on a host machine without spinning up Stylus.
- `shared/abis` is the only place ABIs live. Both `agent/` and `web/` import from it — no manual ABI duplication.
- `solidity/` is a Foundry sub-project because MockOracle is trivial Solidity and Foundry's cheat codes are faster than Stylus for oracle-side tests.

## 3. Smart Contract Design (Stylus / Rust)

### 3.1 Module map

```
crates/vault/src/
├── lib.rs               # #[entrypoint], wires modules
├── storage.rs           # all #[storage] structs + StorageMap layouts
├── deposit.rs           # collateral deposit / withdraw
├── borrow.rs            # borrow / repay
├── lend.rs              # USDC lend / withdraw_lend (ERC-4626 path)
├── liquidate.rs         # liquidation logic
├── health.rs            # health factor + LTV checks
├── interest.rs          # accrual + index update (calls interest-model crate)
├── oracle.rs            # IAggregatorV3-style adapter
├── admin.rs             # owner-only config (asset whitelist, LTV, etc.)
├── events.rs            # Solidity-compatible event topics
└── errors.rs            # typed errors with selector codes
```

Each module is `pub(crate)` and exposes a free function taking `&mut Storage` — *not* methods on the entrypoint struct. The entrypoint is a thin dispatch layer. This keeps each operation independently testable with a fake storage.

### 3.2 Storage layout

Single `#[storage]` struct on the entrypoint, but split into nested `#[storage]` substructs so each module touches only its slice:

```rust
#[entrypoint]
#[storage]
pub struct TesseraVault {
    config:        Config,
    lending:       LendingPool,          // USDC side (ERC-4626 shares)
    collateral:    CollateralBook,       // tStock deposits per user
    debt:          DebtBook,             // borrow positions per user
    interest:      InterestState,        // global accrual indices
    paused:        StorageBool,
}

#[storage]
pub struct Config {
    owner:                 StorageAddress,
    usdc:                  StorageAddress,     // borrow asset
    oracle:                StorageAddress,     // MockOracle / Chainlink
    agent:                 StorageAddress,     // authorised liquidator
    asset_whitelist:       StorageMap<Address, AssetParams>,
}

#[storage]
pub struct AssetParams {
    enabled:               StorageBool,
    max_ltv_bps:           StorageU16,         // e.g. 7000 = 70%
    liquidation_threshold_bps: StorageU16,     // e.g. 8500 = 85%
    liquidation_bonus_bps: StorageU16,         // e.g. 500 = 5%
    price_feed_id:         StorageFixedBytes<32>,
}

#[storage]
pub struct LendingPool {
    total_assets:          StorageU256,        // USDC held + lent
    total_shares:          StorageU256,        // ERC-4626 shares minted to lenders
    shares_of:             StorageMap<Address, StorageU256>,
}

#[storage]
pub struct CollateralBook {
    // user → token → amount
    deposits: StorageMap<Address, StorageMap<Address, StorageU256>>,
    // user's deposited token set, for iteration during health checks
    tokens_of: StorageMap<Address, StorageVec<Address>>,
}

#[storage]
pub struct DebtBook {
    // user → principal USDC (in raw units, no interest)
    principal: StorageMap<Address, StorageU256>,
    // user → interest_index at last update (snapshot)
    user_index: StorageMap<Address, StorageU256>,
}

#[storage]
pub struct InterestState {
    borrow_index:          StorageU256,        // monotonic accumulator, 1e18 base
    last_accrual_ts:       StorageU64,
    base_rate_bps:         StorageU16,
    slope1_bps:            StorageU16,
    slope2_bps:            StorageU16,
    optimal_util_bps:      StorageU16,
}
```

Notes:

- `StorageMap<Address, StorageMap<...>>` is the canonical nested-map pattern in `stylus-sdk` — see `docs/stylus-sdk-rs/examples/mapping`.
- `tokens_of` is needed because Solidity-style mappings have no iteration; we maintain it explicitly on first deposit, and *never shrink it* (cheap empty entries are fine, and shrinking opens reentrancy questions).
- Interest is tracked as a **single global borrow index** scaled by 1e18, à la Compound's `borrowIndex`. Each borrower stores a snapshot; current debt = `principal * current_index / user_index`. This avoids per-user accrual writes on every block.

### 3.3 Public interface

ERC-4626 surface is only on the **lender side** (USDC vault). The collateral side uses bespoke functions because ERC-4626 is single-asset and we accept multiple tokenized stocks.

```rust
// ===== Lender side (ERC-4626 on USDC) =====
fn asset()       -> Address;                  // USDC
fn total_assets()-> U256;
fn deposit(assets: U256, receiver: Address) -> Result<U256, Error>;  // returns shares
fn mint(shares: U256, receiver: Address)    -> Result<U256, Error>;
fn withdraw(assets: U256, receiver: Address, owner: Address) -> Result<U256, Error>;
fn redeem(shares: U256, receiver: Address, owner: Address)   -> Result<U256, Error>;
fn convert_to_shares(assets: U256) -> U256;
fn convert_to_assets(shares: U256) -> U256;

// ===== Borrower side (bespoke) =====
fn deposit_collateral(token: Address, amount: U256) -> Result<(), Error>;
fn withdraw_collateral(token: Address, amount: U256) -> Result<(), Error>;
fn borrow(usdc_amount: U256)  -> Result<(), Error>;
fn repay(usdc_amount: U256)   -> Result<U256, Error>;     // returns remaining debt

// ===== Liquidation =====
fn liquidate(borrower: Address, repay_amount: U256, collateral_token: Address)
    -> Result<U256, Error>;                                // returns seized collateral

// ===== Views =====
fn health_factor(user: Address)   -> Result<U256, Error>; // 1e18-scaled, >1e18 = healthy
fn collateral_value_usd(user: Address) -> Result<U256, Error>;
fn debt_of(user: Address)         -> Result<U256, Error>;
fn utilization_bps()              -> U256;
fn borrow_rate_bps()              -> U256;
fn supply_rate_bps()              -> U256;

// ===== Admin (onlyOwner) =====
fn set_asset_params(token: Address, params: AssetParams);
fn set_oracle(addr: Address);
fn set_agent(addr: Address);
fn pause(); fn unpause();
```

`Error` is a single enum implementing `From<erc4626::Error>` and `From<erc20::Error>` so we can `?` through OZ helpers cleanly. Each variant has a unique 4-byte selector for client decoding.

### 3.4 Math

#### 3.4.1 Health factor

```
collateral_value_usd(u) = Σ (deposits[u][token] * price(token) * liq_threshold_bps / 10000)
debt(u)                 = principal(u) * borrow_index / user_index(u)
health_factor(u)        = collateral_value_usd(u) * 1e18 / debt(u)
```

`debt(u) == 0` is reported as `U256::MAX` (infinitely healthy). The 1e18 scale lets the agent compare against simple integer thresholds (`1.1e18`, `1.05e18`, `1.0e18`).

#### 3.4.2 Interest accrual

Borrow index ticks forward on every state-changing call (`borrow`, `repay`, `liquidate`, `deposit` to lender pool, `withdraw` from lender pool):

```
dt              = now - last_accrual_ts
rate_per_sec    = borrow_rate_bps * 1e18 / (10_000 * SECONDS_PER_YEAR)
new_index       = old_index * (1e18 + rate_per_sec * dt) / 1e18
borrow_index    = new_index
last_accrual_ts = now
```

The interest split: a fixed `reserve_factor_bps` (default 1000 = 10%) goes to a protocol reserve address; the rest flows to lenders via `total_assets`. For MVP, reserve factor is 0 to simplify accounting — we set it before mainnet.

#### 3.4.3 Utilization curve

Lives in `crates/interest-model/src/lib.rs` as a pure function so it can be property-tested without Stylus:

```rust
pub fn borrow_rate_bps(
    util_bps: u32,
    base_bps: u32,
    slope1_bps: u32,
    slope2_bps: u32,
    optimal_util_bps: u32,
) -> u32 {
    if util_bps <= optimal_util_bps {
        base_bps + (util_bps * slope1_bps) / optimal_util_bps
    } else {
        let excess = util_bps - optimal_util_bps;
        let max_excess = 10_000 - optimal_util_bps;
        base_bps + slope1_bps + (excess * slope2_bps) / max_excess
    }
}
```

Defaults (MVP, in bps): `base=200, slope1=400, slope2=6000, optimal=8000`. Yields ~6% borrow at 80% utilization, ~62% at 100%. Supply rate = `borrow_rate * util * (1 - reserve_factor)`.

#### 3.4.4 Liquidation

Standard Aave-style partial liquidation:

```
max_repay      = debt(borrower) / 2                    // 50% close factor
repay          = min(repay_amount, max_repay)
seize_value    = repay * (10000 + bonus_bps) / 10000
seize_amount   = seize_value / price(collateral_token)
```

Liquidator pays USDC, receives the seized stock token. If `seize_amount` exceeds the borrower's deposit of that token, we clamp and the residual debt remains (and the position likely becomes bad debt — flagged as event).

**Bad debt remediation (resolves G10).** If a position is fully drained of collateral and still has `principal > 0`, the vault emits `BadDebtRealized(user, residual_usdc)` and the agent calls `admin.write_off(user)`, which:

1. Zeroes the user's `principal` and `user_index`.
2. Decrements `total_borrows` by the residual.
3. Debits the residual from `total_assets` of the lender pool. *Lenders absorb the loss proportionally to shares* (Aave V2 socialised-loss model). This is acceptable because:
   - We target near-zero bad debt via the 5% liquidation bonus and agent latency under 10s.
   - A reserve factor is configurable (defaults to 0 in MVP, will be > 0 at mainnet) to build a buffer that absorbs losses before lenders.

PRD §10 targets *zero* bad debt; this mechanism is the safety net for the case the agent missed an event. It is *not* the planned operating mode.

### 3.5 Oracle interface

Wraps both MockOracle (testnet) and Chainlink AggregatorV3 (mainnet) behind a single `IPriceOracle` trait used internally:

```rust
pub trait IPriceOracle {
    fn price_usd_8(&self, token: Address) -> Result<U256, Error>;  // 8-decimal price
}
```

Implementation calls into the external oracle contract via `stylus_sdk::call::call`. The 8-decimal convention matches Chainlink, so the swap from mock → real oracle is a single address change.

**Staleness:** any feed older than `MAX_PRICE_AGE` (default 3600s for MVP, will be 60–300s in prod) reverts with `Error::StalePrice`. The agent never sees a stale price drive a liquidation.

### 3.6 Access control

- `owner` — multisig in prod, single EOA in testnet. Configures parameters.
- `agent` — single agent address. Only address allowed to call `liquidate` for the MVP (this is a deliberate trade-off for hackathon timeline; mainnet will open liquidation to any address with a small bond, per Aave's permissionless model).
- Everyone else — vault user.

Modifier helpers in `admin.rs` (`only_owner`, `only_agent`) revert with typed errors.

### 3.7 Events

Mirroring Aave/Compound conventions for indexer compatibility:

```
Deposit(asset, user, amount, shares)
Withdraw(asset, user, amount, shares)
CollateralDeposit(user, token, amount)
CollateralWithdraw(user, token, amount)
Borrow(user, amount, new_principal, new_index)
Repay(user, amount, new_principal)
Liquidate(borrower, liquidator, repay_amount, collateral_token, seize_amount)
AssetConfigured(token, max_ltv, liq_threshold, liq_bonus)
AgentSet(old, new)
Paused(by)  Unpaused(by)
```

## 4. Risk Agent Design (Vibekit / TypeScript)

### 4.1 What the agent actually does in MVP

The PRD lists ambitious AI capabilities (sentiment, hedging, prediction). **None of those ship in week 1–3.** MVP agent has exactly three behaviours, all deterministic with an LLM only on the explanation/alert path:

1. **Poll** vault state every 10s (configurable). Maintain in-memory snapshot of users with `principal > 0`.
2. **Liquidate** when any user's `health_factor < 1e18`. Submit `liquidate(user, repay, token)` transaction.
3. **Alert** through structured logs (JSONL) + write to a `latest_alerts.json` blob the UI reads, when `health_factor < 1.1e18`.

The LLM is used only to:
- Generate human-readable explanations for alerts shown in the UI (`"AAPL dropped 18% in the last hour; your position is approaching liquidation."`).
- Parse the natural-language strategy config from `/agent` page into structured parameter overrides.

### 4.2 Run loop

```ts
// agent/src/index.ts
async function tick() {
  const block = await client.getBlock();
  const users = await indexUsers(block);                  // event-derived
  for (const u of users) {
    const hf = await vault.healthFactor(u);
    if (hf < ONE_E18) await liquidate(u);                 // deterministic
    else if (hf < 1_100_000_000_000_000_000n) alert(u);   // 1.1e18
  }
}
setInterval(tick, 10_000);
```

`indexUsers` reads `Borrow`/`Repay`/`Liquidate` logs since the last checkpoint and maintains a set of users with non-zero debt. We do **not** scan all addresses.

### 4.3 Liquidation transaction safety

- Pre-flight: re-read `healthFactor` immediately before sending tx to avoid racing the user's own `repay`.
- Idempotency: include borrower address + block number in a local SQLite (`better-sqlite3`) table to never re-submit within the same block.
- Funds: agent wallet must hold a small USDC float (we pre-fund with 100 USDC on testnet) to cover `repay_amount`. The seized stock token can be left in the agent wallet for MVP; V2 will auto-swap back to USDC.
- Gas: cap `maxFeePerGas` at 10× the rolling 50-block median to protect against runaway demos.

### 4.4 Failure modes & recovery

| Failure | Detection | Recovery |
|---|---|---|
| RPC down | Tick errors out | Exponential backoff 1s → 60s; alert log entry |
| Tx revert (race with repay) | `eth_call` simulation before send | Skip user this tick |
| Tx revert (insufficient USDC) | Pre-check balance | Log + alert; do not retry until topped up |
| Oracle stale | Vault reverts `StalePrice` | Agent logs and waits — *this is correct behaviour* |
| Agent crash | Process supervisor (`pm2` for demo) | Restart, re-index from checkpoint |

### 4.5 Vibekit usage in MVP (resolves G1)

**Vibekit is the agent framework in MVP, not deferred.** The MVP agent is a Vibekit-registered DeFi agent that exposes Tessera as a protocol the framework can address. Concretely:

- Project depends on `@emberagi/vibekit` (or current package name from `docs/arbitrum-vibekit`).
- We register Tessera as a Vibekit *Protocol* with tool definitions: `getHealth(user)`, `getPositions()`, `liquidate(user, repay, token)`, `getAlerts()`. Vibekit handles the LLM↔tool wiring and observability.
- The **decision logic for liquidation is rules-based, not LLM-decided.** The LLM never authors a `liquidate` call. The framework's tool-call surface is used only by:
  - The `/agent` UI page (user types natural-language strategy → LLM parses to params → stored).
  - The alert-copy generator (rule fires → LLM rewrites JSON into plain-English alert).
- The Trailblazer 2.0 application reuses this same registration; nothing is rebuilt post-hackathon.

If the Vibekit package surface changes meaningfully before week 1, the fallback is a thin Vibekit-shaped wrapper (same tool names, same JSON contract) so the demo and grant narrative are preserved without blocking on upstream.

### 4.6 NL strategy config schema (resolves G7)

User input on `/agent` is parsed by the LLM into this typed struct, validated, and persisted to the agent's SQLite (`agent_config` table, single row, JSON column):

```ts
type AgentConfig = {
  alertThreshold: number;        // HF scaled 1e18, default 1.1e18, range [1.0e18, 2.0e18]
  liquidationThreshold: number;  // HF scaled 1e18, default 1.0e18, fixed in MVP (contract enforces)
  pollIntervalMs: number;        // default 10_000, range [1_000, 60_000]
  paused: boolean;               // hard kill switch
  maxGasGwei: number;            // ceiling on tx submission
  notes: string;                 // free-form, shown back to user for confirmation
};
```

LLM is prompted to emit JSON conforming to this schema; invalid JSON or out-of-range values return an error to the UI without mutating state. Schema is the single source of truth — the contract's behaviour does not change based on this config (the contract enforces liquidationThreshold at `1.0e18`); only the agent's pre-liquidation alerting/throttling does.

### 4.7 Action log API contract (resolves G8)

Agent writes every action as one JSONL line to `agent/logs/actions.jsonl`. A tiny HTTP server on the agent process exposes:

```
GET /actions?limit=50           → most recent N entries (server caps at 200)
GET /alerts/latest              → current open alerts (HF<1.1e18)
GET /health                     → { ok: bool, lastTickAt: iso, errors24h: number }
POST /config                    → AgentConfig (owner-only via shared secret in MVP)
```

Entry schema:

```ts
type Action =
  | { ts: string; kind: "tick"; usersChecked: number; durationMs: number }
  | { ts: string; kind: "alert"; user: Address; hf: string; copy: string }
  | { ts: string; kind: "liquidate"; user: Address; tx: Hex; repay: string; seized: string; token: Address; status: "submitted"|"confirmed"|"reverted" }
  | { ts: string; kind: "error"; where: string; message: string };
```

Retention: file rotates daily, last 7 days kept on disk. UI's `/agent` page calls `GET /actions?limit=50` and renders newest-first.

## 5. Frontend Design (Next.js)

### 5.1 Rendering strategy

- **Default to Server Components** for page shells. They render contract state read on the server via a shared viem public client.
- **Client Components** only for:
  - Wallet connection (`<ConnectKitButton>`)
  - Tx-submitting buttons
  - Live-updating health gauge (subscribes to block via `useWatchBlockNumber`)
- No subgraph. All reads via direct RPC. If RPC pressure becomes a problem we add a thin route handler that caches `total_assets`/`utilization` for 5s using Next's `revalidate`.

### 5.2 The six pages, mapped to PRD §5

| Route | Server fetches | Client interactions |
|---|---|---|
| `/` (dashboard) | `healthFactor`, `collateralValue`, `debt`, agent latest action | Refresh on block; navigate to actions |
| `/deposit` | asset whitelist, current prices, user balances | Token select, amount input, approve+deposit txs |
| `/borrow` | collateral value, current debt, borrow rate | Amount slider, *projected* HF preview, borrow tx |
| `/lend` | lending APY, utilization, user share balance | Deposit/withdraw USDC txs |
| `/agent` | latest 50 actions (from agent's JSONL log surfaced via API route) | NL strategy config (sent to agent's `/config` endpoint), pause toggle |
| `/admin` (resolves G2) | per-user health table, agent status, oracle staleness | **Manual `liquidate` button** per undercollateralized user; emergency `pause()` (owner only) |

`/admin` is gated by checking `connectedAddress == agent || connectedAddress == owner` in a server action. It is the documented mitigation if the agent process is down: any address with the agent or owner key can trigger liquidation manually from the browser. For institutional/DAO users (PRD §4 tertiary persona — resolves G11), `/admin` doubles as a read-only positions table; non-privileged callers see read-only mode with a "Download CSV" export.

### 5.3 Health UI — applying PRD strategic suggestions

Per PRD §1 of the appendix, the dashboard avoids raw `1.07` health factors. The mapping:

| HF (1e18 scale) | Badge | Color | Copy |
|---|---|---|---|
| `>= 1.5e18` | "Safe" | green | "Tessera is watching. No action needed." |
| `1.2 – 1.5e18` | "Healthy" | green | "Comfortable buffer." |
| `1.1 – 1.2e18` | "Watch" | yellow | "Markets are moving — Tessera is monitoring closely." |
| `1.0 – 1.1e18` | "At risk" | orange | "Add collateral or repay to avoid liquidation." |
| `< 1.0e18` | "Liquidating" | red | "Tessera is closing this position to protect you." |

Raw numbers exist behind a "Show advanced" toggle for the DeFi-native secondary persona.

**Portfolio Safety Score (resolves G9).** A single 0–100 number rendered as the headline metric on `/`, mapped from HF so retail users have one thing to look at:

```
score = clamp( round( min(hf / 2e18, 1) * 100 ), 0, 100 )
```

So HF ≥ 2.0e18 → 100; HF = 1.5e18 → 75; HF = 1.2e18 → 60; HF = 1.0e18 → 50; HF < 1.0e18 → < 50 (and at that point the position is being liquidated, so the user mostly sees the "Liquidating" state instead). The number sits next to the badge so the colour carries the action signal and the number carries the magnitude.

### 5.4 Wallet & chain config

- `wagmi` v2 + `viem`. Robinhood Chain custom chain definition lives in `web/lib/chain.ts`.
- ConnectKit for wallet selection. Default to injected + WalletConnect.
- Auto-switch chain if user is on the wrong network; show a clear prompt, not a silent failure.

### 5.5 Type & ABI flow

1. `cargo stylus export-abi` → JSON in `shared/abis/TesseraVault.json`.
2. `wagmi generate` → typed hooks in `web/lib/generated.ts`.
3. UI imports only generated hooks. No raw ABI string literals in components.

## 6. Critical Data Flows

### 6.1 Deposit + borrow (happy path)

```
User                   Web UI                  Wallet                Vault              USDC
  │  click Deposit        │                       │                    │                  │
```

Two transactions (approve + deposit), then one transaction (borrow). We do not attempt EIP-2612 permit for MVP — testnet stock tokens won't have it.

### 6.2 Weekend liquidation (the demo)

```
T+0:  Operator pushes new price to MockOracle (AAPL -20%)
T+2s: Agent's next tick reads vault.healthFactor(alice) = 0.99e18

T+6s: Agent writes log entry; /agent page surfaces "Liquidated position at 12:43:08"
```

End-to-end target latency: under 10 seconds from oracle update to UI confirmation. RPC latency dominates.

## 7. Math, units, and precision

Hard rules to avoid the entire class of DeFi precision bugs:

- All on-chain math in `U256`. Never `u128` for value calculations even if domain fits — multiplications overflow `u128` too easily.
- USD values scaled by 1e8 (Chainlink convention). Token amounts in their native decimals (USDC=6, tStock=18 by default — we assert this in `set_asset_params`).
- Health factor, rates, and indices scaled by 1e18.
- Always multiply before divide. The codebase enforces this via clippy lint + review.
- Round in the protocol's favor for deposits, against for withdrawals — same as OZ ERC-4626 (`docs/rust-contracts-stylus/contracts/src/token/erc20/extensions/erc4626.rs`).

## 8. Security Model

### 8.1 Trust assumptions (MVP, testnet only)

- Oracle is trusted; demo operator can move prices. **Mainnet plan:** real Chainlink + a separate `Sequencer` uptime feed; reject prices during sequencer downtime.
- Agent key is hot; it sits in `.env` on the demo machine. **Mainnet plan:** AWS KMS or Fireblocks; agent only signs `liquidate`; separate cold key controls `owner`.
- Asset whitelist is owner-gated. No permissionless asset listing.

### 8.2 Reentrancy

Every state-mutating external function follows checks-effects-interactions. The only external calls are:
- `IERC20.transferFrom` (collateral in, USDC in) — at the start, *after* state writes that record the credit.
- `IERC20.transfer` (USDC out, collateral out) — at the end.
- `IPriceOracle.price_usd_8` — read-only, but we still treat as untrusted; never branch on intermediate price calls.

We add a simple `nonReentrant` boolean guard in `Config` as belt-and-braces. Stylus's WASM model makes reentrancy harder than EVM but does not eliminate it for cross-contract calls.

### 8.3 Invariants (to be encoded as Foundry/Stylus property tests)

- **I1:** `total_assets >= sum(convert_to_assets(shares_of[u]))` for all `u` (no share inflation).
- **I2:** `sum(principal[u]) <= total_borrows` (no phantom debt).
- **I3:** For every `u`, if `principal[u] > 0` then `collateral_value_usd(u) > 0` (no debt without collateral).
- **I4:** After any successful state mutation, the actor's HF is either ≥ 1e18 *or* the call is `liquidate`/`repay` (no operation can leave a user underwater except liquidation itself, which only does so when the start state was already underwater).
- **I5:** `borrow_index` is monotonically non-decreasing.

### 8.4 Out of scope for MVP

Flashloans, governance, fee switch, supply caps, isolation mode, e-mode. Each is a clearly separable feature that drops in without restructuring storage.

## 14. Glossary

- **LTV (Loan-to-Value):** borrow_value / collateral_value, in bps. We cap at 70% (`max_ltv_bps = 7000`).
- **Liquidation threshold:** the LTV at which liquidation becomes legal. Higher than max_ltv so users have a buffer after borrowing.
- **Health factor:** `collateral_value * liq_threshold / debt`, scaled by 1e18. HF ≥ 1.0e18 means safe.
- **Utilization:** `total_borrows / total_deposits` of the USDC pool. Drives borrow rate.
- **Borrow index:** monotonic accumulator capturing all per-second compounding interest since genesis; lets us avoid per-user accrual.
- **tStock:** any tokenized US equity on Robinhood Chain (tAAPL, tTSLA, tSPY for MVP).
- **Stylus:** Arbitrum's WASM execution environment; lets us run Rust contracts alongside Solidity, with ~10–100× lower gas on math-heavy paths.
- **Vibekit:** EmberAGI's agent framework for Arbitrum DeFi; provides LLM tooling + protocol registration for Trailblazer grants.

## 16. Operator Runbooks (resolves G4)

These are the exact commands the demo operator and on-call engineer execute. Stored as `scripts/runbook/*.ts` so they're version-controlled, not tribal knowledge.

### 16.1 Live demo: the weekend liquidation (PRD §8.3)

Prerequisites: vault + mocks deployed; agent running; alice (demo wallet 1) holds 10 tAAPL; bob (demo wallet 2) holds 5,000 USDC and has lent it.

```bash
# 1. Set starting price (AAPL = $200, position is healthy)
pnpm runbook:price --token=tAAPL --usd=200

# 2. Alice deposits 10 tAAPL and borrows 1,200 USDC (60% LTV → HF ≈ 1.42)
pnpm runbook:demo:setup --user=alice

# 3. Show dashboard: HF "Healthy" (green), Safety Score 71

# 4. Operator: simulate the Saturday gap-down — AAPL drops to $160
pnpm runbook:price --token=tAAPL --usd=160
# Expected: within ~10s, dashboard shows "Watch" (yellow), Score 56
# Expected: /agent log shows alert entry

# 5. Operator: drop further to $140
pnpm runbook:price --token=tAAPL --usd=140
# Expected: within ~10s, agent emits Liquidate event
# Expected: alice's debt halved, alice's tAAPL reduced, agent wallet now holds tAAPL + bonus

# 6. Reset for the next dry run
pnpm runbook:demo:reset
```

`pnpm runbook:demo:reset` redeploys mocks, re-mints, re-funds the agent's USDC float, restarts the agent process via pm2.

### 16.2 Top up the agent's USDC float

When the agent runs low (post-multiple liquidations), it can't pay the next `repay_amount`. Pre-check is part of agent's `/health` endpoint.

```bash
pnpm runbook:agent:topup --usdc=1000
```

Sends 1,000 USDC from the funded deployer wallet to the agent's hot wallet.

### 16.3 Emergency pause

If something is going wrong on testnet (oracle stuck, vault behaving unexpectedly), the owner can pause all state mutations:

```bash
pnpm runbook:vault:pause
# later
pnpm runbook:vault:unpause
```

Both call into the vault's `pause()`/`unpause()` (owner-only). UI detects the paused state via a view function and surfaces a banner.

### 16.4 Recovering from agent crash

```bash
pm2 status                           # see if agent died
pm2 logs tessera-agent --lines 200   # check last error
pm2 restart tessera-agent            # safe to restart; SQLite checkpoint resumes
```

If the SQLite is corrupted (rare), `rm agent/.data/state.sqlite` — agent re-indexes from contract events on next start. Cost: ~1 minute on testnet history.

### 16.5 Cold start of full stack on a new machine

```bash
git clone <repo> && cd tessera
pnpm install && cd contracts && cargo build --target wasm32-unknown-unknown && cd ..
cp .env.example .env                 # then fill in keys
pnpm runbook:deploy --env=testnet    # idempotent
pnpm runbook:agent:start
cd web && pnpm dev
```

## 17. Observability (resolves G5)

PRD §10 demands `agent uptime > 99.5%` and `100% of liquidations succeed → zero bad debt`. Both require measurement.

### 17.1 What we emit

| Surface | Signal | How |
|---|---|---|
| **Vault** | All state changes as events (§3.7) | Indexable; consumed by agent + observability |
| **Agent** | `/health` JSON endpoint | Polled by uptime monitor every 30s |
| **Agent** | `actions.jsonl` | Tailed for liquidation counts, error rates |
| **Agent** | `/metrics` Prometheus endpoint | Counters: `ticks_total`, `liquidations_total`, `errors_total`; gauges: `users_tracked`, `usdc_balance`, `seconds_since_last_tick` |
| **UI** | Browser-side error reporting | Sentry free tier; client errors only |

### 17.2 What we monitor

- **Uptime monitor:** UptimeRobot (free) pings `https://agent.tessera.xyz/health` every 60s. Page on-call if down > 2 min.
- **Liquidation alerter:** a tiny script polls vault events, alerts to Discord webhook whenever `BadDebtRealized` fires (this should be zero — any occurrence is incident-grade).
- **Dashboards:** a single Grafana board (Grafana Cloud free tier) with four panels: agent tick latency, users tracked, USDC float, liquidation count last 24h.

### 17.3 Uptime SLO (PRD §10 target: 99.5%)

`99.5% over 30 days` allows 3.6 hours of downtime. Single Fly.io instance + UptimeRobot is sufficient to *measure* this. Achieving it during the hackathon demo period is the on-call's job (i.e. someone keeps Slack notifications on).

Mainnet plan: two agents in active/standby, leader election via on-chain `agent` address rotation. Out of scope for MVP.
